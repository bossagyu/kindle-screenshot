## パターン / 慣習

### フォローアップ項目のグルーピング提案

レビューで MEDIUM/LOW を多数出した場合、PR 説明や再レビューコメントで PO 向けに「issue A: 性質X / issue B: 性質Y / issue C: 性質Z」のグルーピング案と優先順位（A > B > C）を提示する。これにより PO の起票判断が早くなる。 <!-- Added 2026-05-05 from #1/#2: 再レビューで A/B/C 提案を行ったところ、PO が即座に 3 件起票でき効率的だった -->

### イミュータビリティ規約の確認

`captured_files = captured_files[: -trim]` のようなスライス再代入はイミュータブルパターン（新リスト生成）。`captured_files.pop()` のような mutation を使っていないか確認する。 <!-- Added 2026-05-05 from #1/#2: PR #2 ではイミュータビリティが守られており、レビュー時の良点として評価できた -->

### CRITICAL バグ修正 PR での横断的 grep 確認

CRITICAL バグ修正 PR をレビューする際、**同種の罠が他箇所に残っていないか grep で横断的に確認する**。例: AppleScript 非対応問題なら `grep -rn 'tell application' src/` で全箇所スキャン、CalledProcessError 未捕捉問題なら `grep -rn 'check=True' src/` で確認。レビューコメントで「他に同パターンが残っていないことを確認した」旨も明示的に記載すると、後続の振り返りで効果が見える。 <!-- Added 2026-05-05 from #7/#9: PR #8 のレビュー時に DA が grep で類似パターンを事前確認していれば、`_ACTIVATE_SCRIPT` の同根バグ（issue #9）を 1 PR にまとめて修正できた可能性 -->

## 注意事項

### 自身が作成した PR への APPROVE 制約

GitHub の制限上、自身が作成した PR には APPROVE 不可。COMMENT として「本レビューの判定は APPROVE です。マージしてよい状態です。」を明記する形で対応する。 <!-- Added 2026-05-05 from #1/#2: PR #2 でこの制約に遭遇。今後も同じ運用 -->

### subprocess モックの限界の認識

`unittest.mock.patch("...subprocess.run")` で差し替えるテストは、**Python 側のロジック検証のみ**で、AppleScript やシェルコマンドの構文・互換性は検証できない。レビュー時に「ユニットテスト pass = 動作する」とは決して判断しない。外部 OS API に依存する箇所は、ユニットテスト + 手動統合テストの両方が揃って初めてマージ可。 <!-- Added 2026-05-05 from #7/#9: v0.1 / PR #8 ともにユニットテスト pass で実環境では動かなかった。モックの限界を認識する -->
