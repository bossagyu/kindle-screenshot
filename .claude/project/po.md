## パターン / 慣習

### フォローアップ issue の粒度

DA レビューで MEDIUM/LOW がまとまって出た場合、**性質ごとに 3〜5 件単位でグルーピング**して issue 化する（コード健全性 / パフォーマンス調査 / テスト&ドキュメント など）。1 項目 1 issue は粒度が細かすぎ、全部まとめると粒度が粗すぎる。 <!-- Added 2026-05-05 from #1/#2: PR #2 の DA 再レビューで A/B/C のグルーピングが提案され有効だった -->

### マージ順序の指定

複数のフォローアップ issue が共有ファイル（同一モジュール、README、pyproject.toml など）を変更する場合、**issue 本文の「並行実装の注意」セクションに明示的なマージ順序を記載する**。dev が並行 PR で衝突しないようにするため。 <!-- Added 2026-05-05 from #1/#2: フォローアップ A/B/C で capture.py / pyproject.toml / README に共有部分があり、マージ順序を A→B→C で指定した -->

### CRITICAL バグ対応時の優先度宣言

実機 / 実環境で発覚した CRITICAL バグの issue 起票時、**本文冒頭で「既存フォローアップ #X / #Y / #Z より先行で dev にアサイン」を明示**する。dev / オーケストレーターが優先度を即座に把握できるようにするため。 <!-- Added 2026-05-05 from #7/#9: issue #7 / #9 ともにこの宣言で dev へのアサイン判断が即座にできた -->

## 注意事項

### 振り返りの実施タイミング

PR がマージされたら **その日のうちに** 振り返りを実施し、次の作業に着手する前にロールファイルへの教訓蓄積を完了させる。後回しにすると文脈が薄れて知見が定着しない。 <!-- Added 2026-05-05 from #1/#2: 初回振り返りの実施。プロジェクト初期は学びの密度が高いので即座に蓄積する -->

### 同種バグが連続したらまとめて振り返る

CRITICAL バグが短期間に連続した場合（issue #7 / #9 のように同根の互換性問題が 2 件続く等）、**個別に振り返らずまとめて振り返る** ほうが学びの密度が高い。横断的な傾向（「外部 API 互換性」「ユニットテスト限界」等）が見えやすくなる。 <!-- Added 2026-05-05 from #7/#9: AppleScript 非対応問題を 2 件まとめて振り返ることで、手動統合テスト必須化・モック限界・誤誘導メッセージ等の横断的教訓が抽出できた -->
